# ChronoShift 开发进度流程

## 1. 当前目标

本阶段目标是完成一个稳定可演示的 MVP 闭环：

1. 用户上传短视频。
2. 用户选择目标时代。
3. 后端创建任务并模拟多阶段处理。
4. 前端展示处理进度。
5. 任务完成后展示结构化视频脚本。

当前版本暂不接入真实 ffmpeg、Whisper、大模型、图像生成和 TTS。上传文件仅用于触发演示流程，核心价值是验证产品交互主线和前后端接口形态。

## 2. 已完成能力

- 前端页面：已具备首页上传、时代选择、处理中页面、结果页脚本展示。
- 后端接口：已具备 `/api/upload`、`/api/task/{task_id}`、`/api/result/{task_id}`。
- 预留接口：已保留 BGM 匹配与完整视频合成接口，当前返回占位数据。
- 任务流程：本地演示版本使用后台线程模拟 Celery 任务，避免上传接口阻塞。
- 脚本输出：每个任务生成 3 个 mock 场景，并写入 `outputs/{task_id}/script.json`。
- 文案状态：前端页面、后端错误信息、mock 结果文案已修复为可读中文。

## 3. MVP 开发优先级

### P0：稳定演示闭环

- 保证前端构建通过。
- 保证后端 Python 模块可编译。
- 保证上传后能够看到进度变化，并自动进入结果页。
- 保证结果页能展示时代、台词、时间戳、转场和占位资源。

### P1：真实视频解析

- 使用 ffmpeg 抽取关键帧。
- 使用 ffprobe 获取视频时长、分辨率和帧率。
- 对上传时长、大小和格式做更明确的校验。

### P2：AI 内容理解与时代改写

- 接入 Whisper 或其他 ASR 生成字幕文本。
- 接入多模态模型理解关键帧与字幕。
- 用 LLM 替换当前固定模板，生成稳定的时代化场景、台词和视觉提示词。

### P3：生成能力接入

- 接入图像生成服务，将关键帧重构为目标时代画面。
- 接入 TTS 服务，生成与时代设定匹配的旁白或对白。
- 增加失败重试、降级结果和任务日志。

### P4：后续扩展

- 接入 BGM 匹配或生成。
- 将脚本 JSON、场景图、配音和 BGM 合成为完整视频。
- 增加脚本导出、分享海报、角色卡片和故事摘要。

## 4. 24 小时开发流程建议

| 时间段 | 目标 | 交付物 |
| --- | --- | --- |
| 0-2 小时 | 锁定 MVP 范围 | 明确只做脚本级视频重构，不做逐帧视频生成 |
| 2-5 小时 | 打通前后端闭环 | 上传、任务状态、结果查询接口可用 |
| 5-8 小时 | 稳定 mock pipeline | 可看到解析、理解、改写、生成、打包阶段 |
| 8-12 小时 | 强化结果展示 | 原视频占位、时代化占位、台词、配音、转场清晰展示 |
| 12-18 小时 | 替换第一个真实能力 | 优先接入 ffmpeg 抽帧或 ASR，避免一次接入过多外部依赖 |
| 18-21 小时 | 演示打磨 | 修复文案、错误状态、loading 状态和视觉一致性 |
| 21-24 小时 | 彩排与兜底 | 准备固定 demo 文件、mock 兜底数据和讲解路径 |

## 5. 暂不做范围

- 不做全片逐帧高保真视频生成。
- 不做完整视频合成与渲染。
- 不做 BGM 自动生成。
- 不做复杂实时视频编辑器。
- 不做长视频级别的工业化生产系统。

## 6. 风险与建议

- 外部 AI 服务不稳定：保留 mock 兜底，确保演示不断链。
- 图像与台词风格不一致：先建立固定时代模板，再让 LLM 在模板约束内生成。
- 任务耗时较长：继续使用任务状态轮询，后续可升级为 WebSocket。
- 结果页表现力不足：优先展示清晰脚本结构，再逐步替换真实图片和音频资源。

## 7. 当前验收标准

- `npm run build` 能通过前端构建。
- `python -m compileall app` 能通过后端编译。
- 上传视频后，处理页能显示阶段进度。
- 任务完成后，结果页能展示 3 个 mock 场景。
- 新增文档与产品文档方向一致，且无乱码。
