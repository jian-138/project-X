# ChronoShift 产品文档

## 1. 产品概述

**产品名称：** ChronoShift  
**产品定位：** 面向短视频场景的“平行时代转换器”与“多模态叙事重构引擎”  
**一句话描述：** 用户上传一段抖音短视频，系统自动理解视频内容，并将其转换到古希腊、拿破仑时代、唐朝、赛博朋克等不同世界观中，同时同步重写画面、台词、配音与字幕，生成一个可观看、可传播、可交互的时代化短视频体验。

ChronoShift 的核心不是简单的风格滤镜，而是通过多模态理解与生成能力，让短视频“穿越”到另一个时代，使原始内容获得新的叙事身份、视觉风格和语言表达方式。它面向抖音业务场景，也适用于内容二创、品牌营销、剧情创作、互动娱乐与视觉实验。

---

## 2. 产品背景与机会

短视频平台的内容供给已经高度丰富，但内容创新主要仍停留在剪辑、滤镜、字幕、BGM 等传统手段。随着多模态大模型能力提升，内容生产的边界开始从“编辑视频”转向“重构视频语义”。

ChronoShift 试图回答一个更有想象力的问题：

> 如果一段现代短视频不只是被美化，而是可以被重新放进另一个时代，它会变成什么样？

这类产品有三层价值：

1. **内容创作价值**：为普通用户和创作者提供新的二创方式。  
2. **传播价值**：时代穿越、角色重构、台词改写天然具备强分享属性。  
3. **商业价值**：可承接品牌营销、IP 联动、剧情短片、互动广告与体验式搜索等业务方向。

## 3. 产品目标

### 3.1 业务目标

- 提供一种新的短视频创作方式，让用户“穿越式二创”。
- 为抖音内容生态提供更高感知度的 AI 玩法。
- 探索视觉多模态在体验式搜索、内容重构与互动生成中的应用。

### 3.2 用户目标

- 让普通用户也能快速生成有创意、有故事感的视频版本。
- 让创作者低成本制作更具风格化、可传播的二创内容。
- 让用户在观看视频时获得“原内容被重新解释”的新鲜感。

### 3.3 技术目标

- 完成从视频理解到时代改写的完整多模态链路。
- 在 24 小时黑客松约束下实现可演示、可理解、可传播的 MVP。
- 保持较高的视觉统一性、语义一致性与交互流畅度。

## 4. 核心概念

### 4.1 平行时代转换

将输入视频映射到一个目标时代或文明设定中，例如：

- 古希腊
- 拿破仑时代
- 唐朝
- 中世纪酒馆
- 赛博朋克
- 维多利亚时代

系统会自动完成：

- 场景替换
- 人物身份重写
- 道具与服饰重构
- 台词与字幕时代化
- 配音风格转换
- 背景音乐与氛围重塑（接口预留，MVP 暂不实现）

### 4.2 语义重构

不是简单把原视频套上滤镜，而是保留原始剧情骨架，重新定义视频中的角色、行为、物件和语言，让内容在目标时代中“合理地存在”。

### 4.3 关键帧驱动 + 视频脚本输出

在黑客松场景下，不追求完整的视频合成，而是采用「关键帧图片 + 台词字幕 + 配音音频 + 转场描述」的方式，输出一套可执行的**视频脚本**。脚本包含每个场景的时代化关键帧、对应台词、配音文件和镜头过渡说明，用户或后期工具可据此快速组装成片。

## 6. 核心使用场景

### 场景 A：现代短视频穿越到古希腊

用户上传一段现代探店视频，系统自动将其转换为古希腊市场中的哲学辩论场景，人物穿上长袍，台词改写为古典风格，配音变得更具史诗感。

### 场景 B：普通 vlog 变成拿破仑时代纪实片

用户的日常 vlog 变成 19 世纪欧洲风格纪录片，旁白、人物对白、场景建筑与音乐都同步时代化。

### 场景 C：剧情视频变成中世纪酒馆互动剧

原本的现代剧情视频被改造成酒馆冒险剧情，用户可以继续与视频中的角色互动，推动后续情节发展。

### 场景 D：品牌视频变成穿越广告片

品牌宣传视频被改写为某个时代背景下的“历史广告”，用于活动传播与社交平台扩散。

## 7. 产品形态

ChronoShift 是一个轻量级 Web 产品，核心流程如下：

1. 用户上传短视频。  
2. 选择目标时代或世界观。  
3. 系统分析视频内容并生成时代重构方案。  
4. 输出时代化场景关键帧图片、重写台词与对应配音音频，打包为视频脚本。  
5. 用户查看「原视频 vs 平行时代脚本」的对比结果，按场景浏览图片、台词与配音。  
6. 用户可一键分享生成结果。

## 8. 核心功能

### 8.1 视频上传与解析

- 支持上传常见短视频格式。
- 自动抽取关键帧。
- 自动识别字幕、语音与画面内容。

### 8.2 多模态视频理解

- 识别场景类型。
- 识别人物数量、身份、动作与情绪。
- 提取视频叙事结构。
- 理解对话内容与语义重点。

### 8.3 时代转换引擎

- 根据目标时代生成转换规则。
- 将现代物体、建筑、服饰、称呼替换为时代对应元素。
- 保留原始剧情逻辑，但改变表达方式。

### 8.4 台词与字幕重写

- 将现代口语重写为目标时代语言风格。
- 生成更符合场景氛围的字幕。
- 让视觉内容与语言表达保持一致。

### 8.5 配音重构

- 根据时代生成不同声线与叙述口吻。
- 支持史诗感、纪实感、古典感、酒馆感等风格。

### 8.6 场景图片生成与脚本打包

- 对每个关键帧进行时代化重绘，生成场景图片。
- 将场景图片、对应台词、配音音频、镜头过渡描述打包为视频脚本。
- 输出结构化的脚本 JSON，便于后续组装或导入剪辑工具。

### 8.7 背景音乐匹配（接口预留）

- 提供背景音乐匹配的 API 接口与数据模型定义。
- MVP 阶段返回占位数据，后续版本接入音乐生成或素材库匹配。

### 8.8 视频合成（接口预留）

- 提供将脚本 JSON + 场景图片 + 配音 + BGM 合成为视频的 API 接口定义。
- MVP 阶段仅输出脚本，视频合成在后续版本中实现。

### 8.9 对比展示

- 左侧显示原视频关键帧。
- 右侧显示时代化结果。
- 支持时间轴联动播放。

## 9. 用户流程

### 9.1 标准流程

1. 用户进入首页。  
2. 点击上传视频。  
3. 选择目标时代。  
4. 系统开始分析视频。  
5. 系统展示中间结果：场景识别、人物识别、剧情摘要。  
6. 系统生成时代化台词、关键帧图片与配音。  
7. 用户查看视频脚本（场景图片 + 台词 + 配音）并分享。

### 9.2 高级流程

1. 用户上传视频。  
2. 用户手动选择人物、片段或重点镜头。  
3. 用户选择转换风格：史诗、戏谑、纪录片、酒馆冒险等。  
4. 系统生成不同版本供用户切换。  
5. 用户保存喜欢的版本。

## 11. 技术方案

### 11.1 总体架构

ChronoShift 采用”理解 → 改写 → 生成 → 展示”的流水线结构。

#### 结构分层

- **前端层**：负责上传、时代选择、结果展示与对比交互。  
- **后端层**：负责视频处理、任务调度、文件管理、并发控制。  
- **AI 层**：负责视频理解、台词改写、图像生成、配音生成。  
- **脚本输出层**：负责将场景图片、台词、配音、转场描述打包为结构化视频脚本。

### 11.2 核心模块

#### 视频解析模块

- ffmpeg 抽帧
- Whisper/ASR 识别语音
- 提取时间轴和字幕

#### 多模态理解模块

- 输入关键帧与字幕
- 输出场景、人物、情绪、剧情摘要

#### 时代改写模块

- 由大模型生成时代化世界观与台词
- 生成适配目标时代的视觉提示词

#### 图像生成模块

- 对关键帧进行时代化重绘
- 使用统一风格提示词保证视觉一致性

#### 音频合成模块

- 生成时代化旁白与对白
- 背景音乐匹配接口（MVP 阶段返回占位数据）

#### 视频合成模块（接口预留）

- 定义脚本 JSON → 视频的合成接口与数据模型
- MVP 阶段仅输出脚本，合成逻辑在后续版本实现

#### 视频脚本打包模块

- 将场景关键帧图片、重写台词、配音音频、镜头过渡描述打包为结构化脚本 JSON
- 提供脚本预览与逐场景浏览能力

### 11.3 并发与任务管理

黑客松场景下虽然用户量不大，但单个视频处理链路涉及多轮 AI 模型调用（理解、改写、生图、TTS），每轮耗时从数秒到数十秒不等。需要设计合理的并发策略以保证系统稳定性和用户体验。

#### 任务队列设计

- 采用 **Celery + Redis** 作为任务队列与消息中间件。
- 每个上传视频创建一个主任务，主任务下拆分为多个子任务：
  - 视频解析子任务（抽帧 + ASR）
  - 多模态理解子任务
  - 时代改写子任务
  - 图像生成子任务（多关键帧并行）
  - TTS 配音子任务
  - 脚本打包子任务
- 子任务间存在依赖关系，通过 Celery Canvas（chain / chord / group）编排：
  - `chain(parse, understand, rewrite)` — 串行依赖链
  - `group(img_gen_1, img_gen_2, ..., tts)` — 并行生成组
  - 最终 `chord` 汇聚所有生成结果后触发脚本打包

#### 并发控制策略

- **Worker 并发数**：Celery worker 并发数限制为 `N`（建议 N = CPU 核心数或 GPU 显存允许的并发推理数）。
- **AI API 速率限制**：
  - 对每个外部 API（Gemini/GPT、图像生成、TTS）设置独立的令牌桶限流器。
  - 图像生成 API 并发请求数限制为 2-3（避免触发服务端限流）。
  - 大模型 API 设置 QPM（queries per minute）上限，超出后自动退避重试。
- **上传并发**：同时处理的视频任务数上限为 3-5，新请求超过上限时排队等待。
- **文件存储并发**：对同一任务的文件读写使用异步 IO，避免阻塞 worker。

#### 任务状态与进度追踪

- 使用 Redis 存储任务状态与进度百分比，前端通过轮询或 WebSocket 获取实时进度。
- 任务状态机：
  - `queued` → `parsing` → `understanding` → `rewriting` → `generating` → `packaging` → `done`
  - 任意阶段可转入 `failed`
- 每完成一个子任务，更新进度百分比（各阶段权重见下表）：

| 阶段 | 进度占比 | 说明 |
|------|---------|------|
| 视频解析 | 0% – 15% | 抽帧 + ASR |
| 多模态理解 | 15% – 30% | 场景/人物/剧情识别 |
| 时代改写 | 30% – 45% | 台词 + 视觉提示词生成 |
| 图像生成 | 45% – 80% | 多关键帧并行生成（最耗时） |
| TTS 配音 | 80% – 90% | 台词配音合成 |
| 脚本打包 | 90% – 100% | 组装输出 JSON |

#### 容错与重试

- **自动重试**：AI API 调用失败时自动重试，最多 3 次，指数退避（1s / 2s / 4s）。
- **优雅降级**：
  - 图像生成失败 → 使用原关键帧加风格滤镜作为 fallback。
  - TTS 失败 → 仅输出文本台词，不阻塞脚本打包。
  - 多模态理解失败 → 使用简化版 prompt 重试一次。
- **超时控制**：单个视频任务全局超时 5 分钟，超时后返回已完成的部分结果。
- **死信队列**：重试耗尽后移入死信队列，记录错误日志供排查。

#### 资源释放

- 任务完成后自动清理中间文件（提取的关键帧、临时音频等）。
- Redis 中的任务状态在完成后保留 1 小时后自动过期。
- 生成的脚本 JSON 和场景图片按任务 ID 存储，支持后续查询。

## 17. 风险与应对

### 风险 1：图像生成不稳定

**应对：** 不追求逐帧完整生成，采用关键帧驱动；图像生成失败时使用风格滤镜作为 fallback。

### 风险 2：时代转换不够统一

**应对：** 建立统一的时代模板和规则表，确保道具、语言、服饰一致。

### 风险 3：AI API 调用慢且存在并发限制

**应对：** 采用 Celery 任务队列编排，关键帧图像生成并行化，设置 API 速率限制与自动重试，单任务超时后优雅降级返回部分结果。

### 风险 4：Demo 表达不清晰

**应对：** 强化“上传视频 → 穿越时代 → 看到结果”的直观主线。

## 19. 产品亮点总结

ChronoShift 的亮点不在于把视频“变好看”，而在于把视频“变成另一个文明中的故事”。它将短视频从单纯的观看对象，升级为可重写、可穿越、可互动的多模态内容载体。

这使得它非常适合：

- 抖音的视觉内容生态
- AI 创意玩法展示
- 黑客松现场演示
- 未来内容产品的概念验证

---

## 20. 结语

ChronoShift 不是一个普通的视频工具，而是一个关于“视频如何拥有平行世界”的实验性产品。它的价值在于把多模态大模型的能力转化为用户能直接感知的内容创新体验。

在黑客松语境里，它既有技术感，又有想象力；既能快速落地，又有足够的未来产品叙事空间。
